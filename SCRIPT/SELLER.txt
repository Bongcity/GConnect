좋아, 그러면 **“상품 정보는 네이버에서 가져오기만 하고, 우리가 수정은 안 한다”**를 전제로 해서
헬스 체크 / 커뮤니케이션 히스토리 / 위험 기능 완충장치까지 다 넣은 **최종 셀러(상점 관리자) 사이트 설계**를 정리해볼게.

---

# 1. 상점 관리자(Seller) 사이트 개요

* 도메인: **`https://seller.gconnect.kr`**
* 대상: 네이버 스마트스토어 운영자/담당자
* 목적:

  1. 내 상점 상품이 GConnect/구글에 **잘 연동되고 있는지** 확인
  2. **노출 결과(노출/클릭/CTR)**를 한눈에 보고 보고서처럼 활용
  3. 네이버 연동(크롤링 / API 키) 상태와 오류를 스스로 체크
  4. 요금제 상태, 만료, 업그레이드 여부 관리
  5. 운영팀(Admin)과 메시지/공지 주고받기

> **중요 전제**
> 상품 정보(상품명/가격/설명/이미지)는 **네이버가 원본**이라
> Seller에서는 **읽기만 가능**하고,
> GConnect에서의 **노출 여부(보이게/숨기기)** 정도만 조정 가능하게 간다.

# 3. 주요 화면 설계

## 3-1. 로그인 & 온보딩

### 3-1-1. 로그인 페이지

* 필드: 이메일, 비밀번호
* 버튼: 로그인
* 링크:

  * 비밀번호 재설정
  * (선택) 회원가입/도입 문의는 IR/메인으로 연결

### 3-1-2. 최초 온보딩(선택)

처음 로그인한 상점만 한 번 보여주는 온보딩:

1. 네이버 상점 정보 입력 (상점명, 스마트스토어 URL)
2. 연동 방식 안내

   * 크롤링만 사용 / API 연동(권장) 중 선택 (설명만, 실제 설정은 연동 페이지에서)
3. 현재 요금제/Trial 안내

## 3-2. 대시보드(`/dashboard`)

### 3-2-1. 상단 KPI 카드

* 현재 요금제 + 남은 기간
* 총 상품 수 / GConnect에 인덱싱된 상품 수
* 최근 30일 기준:

  * 노출수 (Impressions)
  * 클릭수 (네이버 이동 수)
  * CTR

### 3-2-2. 트래픽 그래프

* 최근 30일 일별 노출/클릭 라인 그래프
* 기간 선택 탭: 7일 / 30일

### 3-2-3. **헬스 체크 카드(연동 자가 진단)** ✅

가장 중요하게 잘 보이는 위치에 한 카드로 표시:

**예시 카드 구조**

> 제목: “연동 상태 건강 체크(Health Check)”

* 항목들 (체크리스트 스타일):

  * ✅ **네이버 상점 정보 등록 완료**

    * 조건: 상점명 + 스마트스토어 URL + NaverShopId 저장됨
  * ✅ / ⚠️ **최근 24시간 내 상품 동기화 성공**

    * 조건: 크롤링 또는 API 동기화 Job 성공 기록 있음
  * ✅ / ⚠️ **인덱싱률 80% 이상**

    * 조건: “GConnect에 등록된 상품 수 / 네이버 상품 수” 비율
  * ✅ / ⚠️ **API 키 유효 (등록했을 경우)**

    * 조건: API 테스트 최근 성공

* 각 줄 오른쪽에 “자세히 보기” 링크:

  * 연동 상세 페이지(`/integration`)로 이동해 로그/상태 확인

> 느낌: “내 상점이 GConnect랑 잘 연결돼서 잘 돌고 있는지”를
> 초록 체크/노랑 경고 아이콘으로 **한 눈에** 보여주는 카드.

### 3-2-4. 인기 상품 Top 5

* 상품명 / 최근 30일 클릭 수 / CTR
* 클릭 시 상품 상세(읽기 전용) 패널 표시

## 3-3. 상품 관리(`/products`)

> 여기서는 **네이버 상품 정보는 수정하지 않고**,
> “GConnect에서 노출/비노출 여부”만 조정할 수 있게 설계.

### 3-3-1. 상품 리스트

**필터**

* 검색어: 상품명
* 카테고리
* 노출 상태: 노출 / 비노출
* 연동 상태: 정상 / 오류

**테이블 컬럼**

* 썸네일
* 상품명
* 가격
* 카테고리
* 노출 상태 (토글 스위치 – GConnect에서만 숨김)
* 연동 상태(정상/크롤링 오류/API 오류)
* 마지막 동기화 시간
* 최근 30일 클릭 수

**액션**

* 행 클릭 → 우측 슬라이드 패널로 상품 상세 보기(읽기 전용)
* 노출 상태 토글:

  * GConnect 내에서만 숨기는 역할
  * 네이버 실제 상품은 건드리지 않음

### 3-3-2. 상품 상세 패널

* 상품명, 가격, 이미지, 카테고리, 옵션 등 기본 정보(모두 **읽기 전용**)
* GConnect 노출 상태 토글
* 최근 30일 노출/클릭/CTR 간단 그래프
* 연동 정보:

  * 마지막 동기화 시간, 방식(크롤링/API), 최근 오류 메시지(있다면 표시)

> 상품명/가격/설명은 **네이버에서만 수정**하고,
> GConnect는 “보기 + 통계 + 노출 켜고/끄는 정도만” 하면 된다.

## 3-4. 통계/리포트(`/analytics`)

### 3-4-1. 상단 필터

* 기간: 7일 / 30일 / 90일 / 사용자지정
* 그래프: 일별 노출/클릭/CTR

### 3-4-2. 리포트 탭

1. 상품별 리포트
2. 카테고리별 리포트

#### 상품별 리포트

* 표:

  * 상품명, 노출수, 클릭수, CTR
* 정렬: 클릭수/CTR 기준 정렬 가능

#### 카테고리별 리포트

* 카테고리명, 노출수, 클릭수, CTR
* 어떤 카테고리에 힘을 줄지 판단용

> 이 부분은 전적으로 “보고용”이고, 상품 내용 수정은 여기서도 불가.

## 3-5. 네이버 연동(`/integration`)

여기가 Seller 쪽에서 **가장 중요한 핵심 페이지**.

### 3-5-1. 상점 기본 정보 섹션

* 네이버 상점명
* 스마트스토어 URL
* NaverShopId
* (필요 시) Admin이 잠금 걸어두고 상점주는 보기만 하게 할 수도 있음

### 3-5-2. 연동 상태 요약

* 크롤링 상태:

  * 활성/비활성 표시
  * 마지막 크롤링 시간 + 결과(성공/실패)
* API 연동 상태:

  * API 키 등록 여부
  * 마지막 API 테스트 시간 + 결과

아래에서 각각 상세 설정.

---

### 3-5-3. 네이버 상점명/URL 기반 크롤링 설정 ✅

**UI 요소**

* 제목: “상점 크롤링 설정”
* 설명 박스:

  * “네이버 상점명 또는 스마트스토어 URL을 등록하면, GConnect가 해당 상점의 상품을 자동 수집합니다. (상품 정보 원본은 네이버에 그대로 유지됩니다.)”
* 필드:

  * 네이버 상점명
  * 네이버 스마트스토어 URL
* 버튼:

  * “상점 정보 확인”
    → NaverShopId 검증 및 저장
  * “상품 크롤링 요청”
    → 전체/증분 크롤링 Job 생성 (선택 옵션)

**하단 리스트: 최근 크롤링 내역**

* 실행 시각
* 타입(전체/증분)
* 처리 상품 수
* 결과(성공/실패)
* 실패 시 에러 요약 메시지

---

### 3-5-4. 상점 API 키 연동 설정 ✅

**UI 요소**

* 제목: “API 연동 (권장)”
* 설명:

  * “API 키를 등록하면, 크롤링보다 안정적으로 상품 정보를 동기화할 수 있습니다. 상품 내용은 네이버에서만 수정되며, GConnect는 그대로 반영합니다.”
* 필드:

  * API Key
  * (필요 시) Client ID / Secret 등
* 버튼:

  * “API 연결 테스트”
    → 예: 상품 1개 불러와서 성공/실패 확인
  * “상품 동기화 실행”
    → API 기반 동기화 Job 생성

**상태 표시**

* 현재 API 키 등록 여부
* 마지막 API 테스트 시간/결과
* 최근 API 동기화 Job 결과 요약

### 3-8-2. 1:1 문의/지원 티켓

### 3-8-3. **Admin–Seller 커뮤니케이션 히스토리** ✅

**리스트 예시**

* 각 행:

  * 날짜/시간
  * 발신자: “운영팀(Admin)” / “내 계정”
  * 유형:

    * 공지/안내
    * 티켓 답변
    * 중요한 공지(요금/정책 변화 등)
  * 내용 요약 (클릭 시 전체 보기)

**동작 개념**

* Admin 사이트에서 상점에 남긴 메모 중
  “상점에게 공개” 플래그가 있는 메모 →
  여기 히스토리에 “운영팀 메시지”로 보임.
* 상점주가 Support에서 문의를 남기고,
  Admin이 답변하면 → 히스토리에도 쌓임.

> 결과적으로:
>
> * Admin 쪽 메모 + Seller 쪽 문의/답변이 **한 타임라인에 정리**되면서
> * “언제 누가 무슨 이야기를 했는지” 한눈에 볼 수 있음.

# 6. “위험 기능”에 대한 완충 장치(셀러 측) ✅

Seller는 Admin보다 권한이 적지만, 그래도 몇 가지는 위험함:

* API 키 삭제
* 네이버 연동 완전 해제
* 대량 비노출 처리 (상품 여러 개를 한 번에 숨김)
* 플랜 변경/해지 요청

### 6-1. 공통 보호 UX

1. **확인 모달 필수**

   * 예: “API 키를 삭제하면, 더 이상 자동 동기화가 되지 않습니다. 계속하시겠습니까?”
2. **상점명 입력 요구 (선택)**

   * 진짜 위험한 작업(연동 해제)일 경우:

     * “아래에 상점명을 정확히 입력해 주세요: [상점명]”
3. **Undo 기회 (가능하면)**

   * 대량 비노출 같은 건 작업 직후 상단에 “되돌리기” 버튼 잠시 노출

### 6-2. 구체 적용 예시

* **API 키 삭제 버튼**

  * 바로 삭제 ❌
  * → 모달: “정말 삭제할까요? GConnect는 더 이상 API로 상품을 가져올 수 없습니다.”
  * 확인 후에야 삭제 수행

* **연동 해제(크롤링 중지)**

  * 모달 + 간단한 설명:

    * “연동을 해제해도, 이미 인덱싱된 상품은 당분간 검색에 남을 수 있습니다.”

* **요금제 다운그레이드/해지 요청**

  * “현재 상품 수가 Pro 플랜 기준을 초과하면, 다운그레이드가 제한될 수 있습니다.”
  * 다운그레이드 요청은 ‘요청’만 하고 실제 변경은 Admin이 처리하도록 해도 됨.
